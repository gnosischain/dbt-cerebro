int_consensus_validators_income_daily

Daily per-(date, validator_index) consensus income fact, including exited and zero-balance validators. Amount columns in GNO. Identifier columns (pubkey, withdrawal_credentials, withdrawal_address, status) are joined at the API layer from int_consensus_validators_snapshots_daily to keep this table memory-efficient. Two deposit numbers are emitted because of the post-Pectra accounting problem: pre-Pectra, beacon-chain deposits credited the same day they were requested. Post EIP-7002 / EIP-7251 (MaxEB) a single execution-request can ask for up to 2048 GNO for one validator, but the beacon chain credits it gradually through the pending_deposits queue (churn-bounded), so a 1767-GNO request can land as ~0.01 GNO on the request day and ~1 GNO/day for weeks afterward. The naive income = balance_delta - reported_deposit + withdrawals - cons_net then produces catastrophic per-day errors (~-1767 GNO on request day, large false positives on drain days). Fix: spec-bounded effective-credit. The consensus spec gives a closed-form maximum reward per epoch per validator, base_reward_per_epoch = effective_balance_gwei * BASE_REWARD_FACTOR / sqrt(total_active_effective_balance_gwei) With Gnosis constants BASE_REWARD_FACTOR=25, SLOTS_PER_EPOCH=16, SECONDS_PER_SLOT=5 (so EPOCHS_PER_DAY = 86400 / (5*16) = 1080) and a 3x safety multiplier for attestation + proposer + sync duties, we get expected_reward_cap = 3 * effective_balance_gwei * 25 * 1080 / sqrt(network_active_effective_balance_gwei) (in gwei) effective_deposits_credited = GREATEST( 0, balance_delta + withdrawals - cons_net - expected_reward_cap ) consensus_income_amount_gno = balance_delta - effective_deposits_credited + withdrawals - cons_net This caps daily income at the spec maximum, keeps cumulative balance-flow exact, leaves queued-but-not-yet-credited deposits in reported_deposit until the balance actually moves, and scales with both validator size and total network stake. Same base_reward * sqrt(total_stake)^-1 relation used by beaconcha.in's calculator, applied here as a validity bound on observed income rather than a forward projection. NOT additionally capped at reported_deposit: post-Pectra, on a queue-drain day the reported amount is 0 but the balance jumps by hundreds of GNO; capping at reported would falsely attribute the jump to income. The spec cap alone is sufficient. Edge cases handled: pure reward day, reported-but-not-credited, queue-drain, slashing, and consolidation source/target all reconcile correctly. Cumulatives use effective credits (cumulative_deposits_gno) for total_income_estimated_gno; cumulative_reported_deposits_gno is kept alongside for audit. Incremental strategy follows int_execution_tokens_balances_native_daily — daily deltas + balance reads are pulled only from the incremental window, with prior-state (last balance and cumulatives at the previous partition boundary) loaded from so lifetime cumulatives stay correct without scanning history. balance_prev_gno uses lagInFrame within the window, falling back to prev_state at the partition boundary. Known upstream limitation: consensus.validators crawler outages (observed 2025-12-10 .. 2025-12-28, 20 fully-empty days) cause withdrawals/deposits/ consolidations during the gap to have no matching snapshot, producing a large one-day negative income on the resume day for affected validators. Documented here so consumers know those single-day outliers are upstream-data, not formula.
