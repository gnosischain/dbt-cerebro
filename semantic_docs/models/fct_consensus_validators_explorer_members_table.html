fct_consensus_validators_explorer_members_table

Members table for the Validator Explorer tab: one row per validator under the selected withdrawal credential. Reads int tables directly (cheap: ~558k status rows + ~558k latest-day income rows) rather than routing through api_consensus_validators_performance_latest. v3 additions: * Lifecycle columns: activation_eligibility_epoch, activation_epoch, exit_epoch, withdrawable_epoch, slashed, plus derived activation_date / exit_date / withdrawable_date. * Per-validator balance-weighted 30-day APY: apy_30d = SUMIf(income, eff_balance > 0) / NULLIF(AVGIf(eff_balance, eff_balance > 0), 0) * 365 * 100 with zero-effective-balance rows excluded so idle slots don't pull the mean. * Driver is fct_consensus_validators_status_latest with LEFT JOIN into income/rewards so a validator still pending activation (no income rows) still appears as a row with status='pending_*' and zeros elsewhere — no silent row-drops on the join. FAR_FUTURE_EPOCH sentinel (2^64 - 1 = 18446744073709551615) is emitted by the beacon state for epochs that have not happened yet; these are NULLed out before converting to dates so the UI renders a blank instead of a year in 292 billion AD. Otherwise, derived dates are toDate(genesis_time_unix + epoch * slots_per_epoch * seconds_per_slot).
