int_execution_gpay_user_identity_bridge

INTERNAL ONLY — GP-side analogue of `int_execution_gnosis_app_user_identity_bridge`. Holds raw GP addresses (Safe contracts, initial-owner EOAs, delegate EOAs) paired with their `user_pseudonym` and `identity_role`. The only place in the warehouse where GP raw addresses live alongside pseudonyms — downstream GP models read from this bridge and never recompute the hash. Three identity roles are emitted per Safe (UNION ALL of three CTEs): initial_owner — the EOA that signed the Safe setup (event_kind = `safe_setup` in `int_execution_safes_owner_events`). The most likely value the GP frontend passes to `mixpanel.identify(...)` after onboarding even though the Safe ownership transfers to the sentinel address afterwards. delegate — a spender delegate EOA from `int_execution_gpay_spender_delegates_current`. Usually a backend card-flow key; some flows still identify with it. safe_self — the Safe smart-account address itself, hashed with the same macro. Some GP frontend flows call `mixpanel.identify(safeAddress)` instead of the EOA, so this row is needed to land those events. A 2-owner Safe with 1 delegate produces 4 rows (2 initial_owner, 1 delegate, 1 safe_self). Privacy posture: same as the GA bridge — `internal_only` tag, `expose_to_mcp = false`, no `api:<resource>` tag, `INTERNAL_ONLY_TABLES` denied at MCP runtime. `pseudonymize_address` is deterministic with the same salt (`CEREBRO_PII_SALT`) as the GA bridge and `stg_mixpanel_ga__events`, so when the underlying EOA is the same, the `user_pseudonym` joins on equality across all three sources. This equality is the cross-domain join key for unified MMM + MTA reports.
