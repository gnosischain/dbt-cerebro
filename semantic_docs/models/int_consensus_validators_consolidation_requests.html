int_consensus_validators_consolidation_requests

Deduplicated EIP-7251 (MaxEB) validator consolidation requests. One row per unique (source_pubkey, is_self_consolidation) tuple, keeping the earliest submitted request_slot. is_self_consolidation is derived from (source_pubkey == target_pubkey). The dedup matters because the raw consensus.execution_requests.payload.consolidations array records every operator resubmission, while the beacon chain only processes the first valid request per source (and rejects later resubmissions). On heavy consolidation days this inflated downstream target_inflow_gno by up to 8× before the fix landed. A validator may legitimately have BOTH a self-consolidation (credential switch 0x01 → 0x02) and a later cross-consolidation (balance transfer out, validator exits), which is why dedup partitions by (source, is_self) rather than by source alone.
