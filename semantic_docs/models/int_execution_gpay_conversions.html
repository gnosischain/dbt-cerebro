int_execution_gpay_conversions

GP conversion registry. One row per (conversion event, identity_role) pair via the bridge fan-out — every chain conversion on a GP Safe produces one row per pseudonym × role tied to that Safe. Persona picks the analytical grain at query time: WHERE conversion_kind='gpay_payment' AND identity_role='initial_owner' → owner grain (one human → one row per payment from any owned Safe) WHERE conversion_kind='gpay_payment' AND identity_role='safe_self' → treasury grain (one Safe → one row per payment) WHERE conversion_kind='gpay_payment' AND identity_role='delegate' → operator grain (rarely useful; delegates are usually backend keys) v1 conversion sources: gpay_payment — int_execution_gpay_activity WHERE action='Payment' gpay_cashback_claim — int_execution_gpay_activity WHERE action='Cashback' gpay_funded — first inflow (`Fiat Top Up` or `Crypto Deposit`) into a Safe across the FULL history. Computed via per-Safe `argMin`/`min` over all-time activity (not microbatch-bounded — "first" is global) and then filtered to rows whose first_inflow_ts falls in the target window. `int_execution_gpay_activity` has no `log_index`, so `conversion_dedup_key` is built from (transaction_hash, token_address, counterparty, direction, identity_role) — the activity table's natural unique tuple plus the role fan-out.
