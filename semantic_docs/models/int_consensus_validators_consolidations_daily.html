int_consensus_validators_consolidations_daily

EIP-7251 (MaxEB) consolidation events at per-(date, validator_index, role) grain. See https://notes.ethereum.org/@fradamt/maxeb-consolidation for the spec. Materialized as a table because consolidations are rare and the application day can lag the request by weeks, breaking monthly-partitioned incremental assumptions. Two flavours emitted: * Self-consolidation (source_pubkey == target_pubkey): a credential switch 0x01 -> 0x02 with no balance transfer. Emitted on the request day with role='self', transferred_amount_gno = 0. * Cross-consolidation (source != target): processed at the source's withdrawable epoch — the source's full effective balance transfers to the target and the source exits. Emitted on the inferred APPLICATION day for both source (role='source') and target (role='target'); if application has not yet occurred no row is emitted (a later run will pick it up). Application day is inferred as the first day at/after the request where the source validator's effective_balance_gwei first reaches 0. transferred_amount_gno is taken as the source's last non-zero effective balance, observed regardless of whether that day is before or after the request day — Gnosis's high churn limit means a consolidation can apply in the same slot as the request, leaving no post-request eb>0 row to capture. Pubkey -> validator_index resolution reads fct_consensus_validators_status_latest (not int_consensus_validators_labels) so exited source validators — by definition every successful consolidation source — remain resolvable. JOINs are pre-filtered to only the pubkeys appearing in the current batch's requests, and the source snapshot lookup is bounded to a tight window (-7d/+30d around request_date) so the build doesn't OOM on busy months. v4 (2026-04): final aggregation grain matches the ReplacingMergeTree unique_key (date, validator_index, role). When N sources consolidate into the same target on the same application day the per-counterparty fidelity is collapsed — the target row carries transferred_amount_gno = SUM(incoming), cnt = N, and counterparty_validator_index = any() representative. Before this fix, emitting one row per (date, validator_index, role, counterparty) made all but one of the N target rows get silently dropped by ReplacingMergeTree dedup, under-crediting the target by (N-1)*32 GNO and corrupting per-validator income in int_consensus_validators_income_daily. Mass-balance invariant SUMIf(transferred_amount_gno, role='source') = SUMIf(transferred_amount_gno, role='target') per date is enforced by tests/consensus_consolidations_mass_balance.sql. v5 (2026-04): transferred_amount_gno is the source's last-observed REAL balance (balance_gwei / 1e9), not its effective balance. Effective balance is a 32-GNO-step function that over-reports the actual mass moved when a source has accrued inactivity penalties; using real balance makes the target's consolidation_inflow_gno match the target's actual balance delta, eliminating the ~10-120 GNO per-target "loss" days seen under v4 on heavy-consolidation events. After this change, the target's consensus_income_amount_gno on the application day is ~0 (pure reward) rather than slightly negative. The sweep still uses effective_balance_gwei=0 as the application-day indicator.
