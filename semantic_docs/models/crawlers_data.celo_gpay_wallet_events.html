crawlers_data.celo_gpay_wallet_events

Unified, append-only wallet-lifecycle event log for Celo GP card Safes: issued_at (initial owner(s) at SafeSetup, one row per owner) + add_owner + remove_owner. Replaces the earlier separate celo_gpay_wallets (a per-safe snapshot table whose is_activated/first_spend_at/owner_address MUTATED in place on every daily refresh) and celo_gpay_owner_events tables — feedback from review was that any source where rows change value over time (not append-only) will break incremental models built on top of it, and this is not a hypothetical: an incremental model that already processed a row before a late-arriving fact caught up has no mechanism to revisit it. An append-only log has no row that can ever change, so it's immune to that failure class by construction rather than by careful, easy-to-forget engineering discipline (full-rebuild wallets + wide-enough incremental windows) elsewhere in the pipeline. first_spend_at/is_activated are deliberately NOT events here — derived in dbt directly from crawlers_data.celo_gpay_transfers instead (the fact is already fully present there — anything sent to the settlement address, the same definition the Dune spine itself uses for its own activation signal — so ingesting it separately would just store the same fact twice). Discovered while building this: GP's Celo provisioning flow commonly swaps a Safe's owner within seconds of SafeSetup —in the simplest observed case, the real EOA owner is removed and replaced with a hardcoded dead/sentinel address (0x...002), permanently disabling owner-based signing so only the Roles/Delay module can move funds; more complex multi-owner reconfigurations (1 remove + multiple adds) have also been observed, so this is real variation worth tracking as data, not a single deterministic rule to hardcode. int_celo_gpay_wallets.owner_address (identity) reads only the issued_at events and intentionally never reflects this later churn — mirrors Gnosis Chain's initial_owner vs current-owner split exactly (int_execution_gpay_user_identity_bridge vs int_execution_gpay_wallet_owners). Deliberately minimal columns — no tx_hash/log_index, since nothing downstream reads them; the current-owners fold's same-timestamp tie-break is resolved entirely in dbt (see int_celo_gpay_safe_current_owners.sql) without needing raw chain log metadata. Ingested by click-runner: a one-time full backfill since launch plus a daily 3-day-lookback incremental job (tolerates the job not running for up to 2 days straight), same shape as celo_gpay_transfers.
