int_execution_gpay_safe_identities

PURPOSE ------- This model is the privacy boundary for the Gnosis Pay ↔ Mixpanel bridge. It produces a lookup table from `user_pseudonym` (a salted keyed hash of an on-chain address) to `gp_safe` + `identity_role`, so the downstream Mixpanel fact model can join on `mp.user_id_hash = id.user_pseudonym` without ever touching a raw address. Everything upstream (int_execution_safes_owner_events, int_execution_gpay_spender_delegates_current, int_execution_gpay_wallets) still holds raw addresses because the on-chain pipeline needs them for joins. Everything downstream (fct_mixpanel_ga_gpay_users, fct_mixpanel_ga_gpay_crossdomain_daily) only sees `user_pseudonym` and `gp_safe`. This model is the one place where the `pseudonymize_address` macro is applied; the macro is the same one stg_mixpanel_ga__events uses at ingest, and the salt is the same env var (CEREBRO_PII_SALT), so the join keys are identical on both sides. WHY THREE IDENTITY ROLES ------------------------ The same person shows up in Mixpanel as one of THREE different identifiers depending on which GP frontend flow created their session. Picking only one of them collapses the match rate to near zero. The three roles, in order of likely match volume: initial_owner — the EOA the user signed in with via passkey or wallet during onboarding. After the GP setup ceremony, ownership of the Safe is transferred to the inaccessible sentinel 0x...0002 and this EOA only retains access through the Delay module. But it is still whatever wallet the user authenticated with, so it is the most likely value the GP frontend passes to mixpanel.identify(...). Pulled from int_execution_safes_owner_events where event_kind='safe_setup'. delegate — a spender delegate EOA assigned via Roles v2 AssignRoles events. These are usually backend- managed card-flow keys, but a few flows may identify with them. Pulled from int_execution_gpay_spender_delegates_current. safe_self — the Safe smart-account address itself. Some GP frontend flows call mixpanel.identify(safeAddress) instead of identify(eoaAddress). Pulled directly from int_execution_gpay_wallets. Row grain: one row per (gp_safe, identity_role, user_pseudonym). A Safe with two initial owners and three delegates produces 6 rows (2 + 3 + 1 self). A single user_pseudonym can appear under multiple gp_safe values if the same person owns multiple GP cards. PRIVACY INVARIANT ----------------- Every column derived from a wallet address is hashed via pseudonymize_address. The output schema contains user_pseudonym (irreversible without the salt) and gp_safe (treated as a smart- account identifier rather than PII — flip if that classification ever changes). Raw EOA addresses never reach this model's output.
