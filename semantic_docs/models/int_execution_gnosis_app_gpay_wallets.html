int_execution_gnosis_app_gpay_wallets

Gnosis Pay wallets (Safes) that have been or currently are controlled by a Gnosis App user, via the Safe's Zodiac **Delay Module**. Heuristic (Hugo's Dune query 6675111, Bernardo review April 2026): The Delay Module bound to each GP Safe has its OWN access control — Zodiac's enableModule(address)/disableModule(address) functions, which emit EnabledModule/DisabledModule events. Any address enabled as a module on the Delay can queue transactions on the GP Safe. A GP wallet is "on Gnosis App" iff a GA user address has been enabled (net +1) as a module on its Delay. Events scanned (from execution.logs): EnabledModule topic0 = 0xecdf3a3e…f8440 (module addr in DATA field) DisabledModule topic0 = 0xaab4fa2b…54276 (module addr in DATA field) Mapping Delay Module → GP Safe comes from int_execution_gpay_safe_modules.contract_type = 'DelayModule'. Incremental: on each run, scans only delay modules that had an EnabledModule/DisabledModule event in the last 30 days (or since previous-run watermark), then recomputes the FULL event history for just those modules. delete+insert on unique_key=pay_wallet keeps unchanged rows from previous runs. Full-refresh scans everything back to 2023-06-01. Onboarding classes (per Daniel's dashboard request): onboarded_via_ga : the very first module ever enabled on this Delay is a GA user — user went from zero → Gnosis Pay via Gnosis App. imported : a GA user was added as a module later — user already had Gnosis Pay, then linked their GA Safe. One row per GP Safe. A Safe with multiple GA modules enabled is still counted once.
