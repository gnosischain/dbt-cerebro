int_celo_gpay_activity

Per-transfer classified Celo GP activity. Classification happens HERE, not upstream in Dune — Dune only supplies raw whitelisted-token transfers touching a GP Safe (crawlers_data.celo_gpay_transfers: sender, receiver, token, amount). This mirrors int_execution_gpay_activity on Gnosis Chain, which also classifies from raw log data rather than a pre-classified source, so fixing/extending classification is a dbt edit + re-run here, not a Dune query edit + re-backfill. Actions: Payment (Safe -> AggregateBridge settlement), Withdrawal (Safe -> anything else, including hypothetically another GP Safe), Reversal (AggregateBridge -> Safe; a processor-side refund of a failed/disputed charge), Top-up (anything else -> Safe, e.g. a MiniPay funding wallet). No Cashback action exists; the GNO cashback program is Gnosis Chain only. CASE ordering in the model checks sender-based branches (Payment/ Withdrawal) before receiver-based branches (Reversal/Top-up) — required so a hypothetical Safe-to-Safe transfer classifies as a Withdrawal from the sending Safe, not a Top-up to the receiving Safe (safe_address always resolves to sender when sender is a Safe, so action must agree). Rows where neither sender nor receiver matches int_celo_gpay_wallets are dropped, not guessed at — observed in practice (2026-07-02 test backfill, 3 rows) as a real race between crawlers_data.celo_gpay_transfers and int_celo_gpay_wallets, which are each independently derived from the same live, continuously-indexing Dune spine rather than one consistent snapshot: a brand-new Safe's transfer can be extracted moments before that same Safe lands in the wallets snapshot. materialized='incremental' with incremental_strategy='insert_overwrite' and apply_monthly_incremental_filter (same macro/pattern already used by Gnosis Chain's int_execution_gpay_activity_daily and int_execution_gpay_wallet_owners) — recomputes the whole current calendar-month partition every run, not just literally-new rows. This is what preserves the self-healing behavior above under incremental materialization: a Safe recognized a few seconds/minutes late (the only lag actually observed) gets correctly reclassified on the next run as long as it's within the same month. int_celo_gpay_wallets itself stays a full rebuild regardless (see that model's own doc) — only this per-transfer model needs incrementality, since it's the one whose row count scales with transaction volume, not signup volume.
