fct_consensus_validators_explorer_latest

Per-operator (withdrawal_credentials) latest-snapshot roll-up. Feeds the KPI cards on the Validator Explorer tab. Reads the int tables directly (not the composite api_consensus_validators_performance_latest view) so the outer GROUP BY across ~558k validators stays under the cluster's memory cap. v3 additions: lifecycle counts (active_count / exited_count / pending_count / slashed_count) classified from validator status (active_ongoing/active_exiting/ active_slashed -> active; withdrawal_done/withdrawal_possible/exited_* -> exited; pending_* -> pending; slashed flag -> slashed); balance-weighted credential-level 30-day APY computed once per credential as apy_30d = SUM(income_30d) / AVG(effective_balance_30d) * 365 * 100, with rows where effective_balance = 0 excluded so idle slots don't drag the mean down. Driver is fct_consensus_validators_status_latest via LEFT JOIN so freshly-activated or never-activated validators with no income rows still count toward validator_count and populate status fields. Defensively accepts both Gnosis-style status values and Ethereum-mainnet-style (active_exiting, active_slashed, exited_*) in case the upstream crawler starts emitting them.
