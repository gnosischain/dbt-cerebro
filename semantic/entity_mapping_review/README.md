# Entity mapping review

Bottom-up entity (join-key) mapping proposed for **every dbt model** (1,258 total),
generated by live-profiling the warehouse. Nothing here is applied — it awaits sign-off.

## Files
- `proposed_dictionary_additions.yml` — the actionable proposal: new entities, new
  column aliases for existing entities, and entityless-scanner broadening. START HERE.
- `review.html` — open in a browser: filterable per-model explorer (search, filter by
  module / entity / role / confidence) + the proposed changes rendered as tables.
- `per_model_mappings.json` — machine-readable: every model with its column->entity
  mappings (entity, role primary/foreign, confidence, distinct_count, sample_value, note).
- `expansion_raw.json` — the aggregated new-entities / column-additions / misses.

## How it was produced
- 536 identifier-bearing models mapped (390 **live-profiled** against ClickHouse with
  bounded 100k-row samples; 146 `api_*` views mapped from schema — querying those views
  OOMs the shared cluster). 722 models are entityless (pure aggregates); a 45-model
  sample was swept for missed keys.
- 1,040 column->entity mappings, 706 high-confidence. Roles: 106 primary / 934 foreign.

## Decisions needed (before regenerating the entity overlay)
1. **`address` role policy** — ~30 role-variant columns (from/to/sender/owner/recipient/
   withdrawal_address/ubo_address/...) all resolve to the address key space, but a single
   transfer row carries several. Decide: one `address` entity with per-column roles, or
   split senders/receivers, before folding these aliases in.
2. **New entities** — keep/merge/drop the 7 candidates (merge `offer` into `token_offer`;
   reconsider `safe_module` sensitivity; `bls_withdrawal_pubkey` is a single model).
3. **`label` handling** — overloaded across token/pool/project/symbol; needs per-model
   mapping, not a scanner rule.

Once decided, fold choices into `../entity_dictionary.yml`, rerun
`scripts/semantic/generate_entities.py`, and rebuild the registry.
