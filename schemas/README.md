# `schemas/` — published contracts

This directory holds **hand-authored JSON Schemas** that act as the versioned,
cross-repo contract between `dbt-cerebro` (which *produces* data) and downstream
consumers such as `cerebro-mcp` and external tools (which *consume* it).

Nothing in here is generated. Each file is a source-of-truth contract that is
committed, published to GitHub Pages, and validated against the actual build
output by the test suite.

| File | Contract for |
| --- | --- |
| [`semantic_graph_catalog.schema.json`](semantic_graph_catalog.schema.json) | the published knowledge-graph catalog (`target/semantic_graph_catalog.json`) |

---

## `semantic_graph_catalog.schema.json`

### What it is

The formal spec (JSON Schema Draft 2020-12) for the **knowledge-graph catalog**
— the artifact that turns dbt models carrying a `config.meta.cerebro.graph`
block into a typed node/edge graph plus a search corpus, consumed by the Graph
Explorer mini-app and the graph MCP tools (`search_graph_catalog`,
`explore_neighborhood`, `calculate_flow_efficiency`).

Do **not** confuse the two same-named files:

| File | Role | Origin | In git? |
| --- | --- | --- | --- |
| `schemas/semantic_graph_catalog.schema.json` | the **schema** (this contract) | hand-authored | **yes — commit it** |
| `target/semantic_graph_catalog.json` | the **instance** (the data) | generated by `build_registry.py` | no — `target/` build artifact |

The schema is published at a stable URL (its `$id`):
`https://gnosischain.github.io/dbt-cerebro/schemas/semantic_graph_catalog.schema.json`
— which only works because it is committed and served from gh-pages.

### The pipeline

```
config.meta.cerebro.graph   (authored in semantic/authoring/**/semantic_models.yml)
        │
        ▼  build_registry.py  → flattened into model.semantic.meta.graph in target/semantic_registry.json
        ▼  build_registry.build_graph_catalog()  (scripts/semantic/build_registry.py:807)
        ▼
target/semantic_graph_catalog.json           ← the INSTANCE (self-hashed, deterministic)
        │
        ├─ validated (structure) against ── schemas/semantic_graph_catalog.schema.json   ← THIS FILE
        │      via tests/test_graph_catalog.py  (jsonschema.validate)
        ├─ validated (authoring errors) by ─ scripts/semantic/graph_gate.py  (blocking CI gate)
        │
        ▼  loaded by cerebro-mcp  → GraphProfile dataclass + Graph Explorer + graph MCP tools
```

### Sections (top-level keys, all `required`)

| Key | Contents |
| --- | --- |
| `metadata` | `schema_version`, content hashes (`graph_catalog_hash`, `registry_manifest_hash`), counts |
| `node_types` | entity kinds from [`semantic/graph_kinds.yml`](../semantic/graph_kinds.yml) (address, token, user_pseudonym, …) |
| `edge_types` | typed relationships derived from the `graph` blocks |
| `profiles` | the graph-edge definitions — **strict 1:1 with the cerebro-mcp `GraphProfile` dataclass** |
| `join_edges` | cross-sector relationships (`semantic/relationships/*.yml`) |
| `metric_bindings` | which metrics attach to which edge/node types |
| `search_documents` | the BM25 search corpus for `search_graph_catalog` |

`additionalProperties: false` is set throughout — an unexpected key **fails
validation**. That strictness is what makes the contract trustworthy.

## How it is maintained

The `profiles` object is the load-bearing part, and its field set lives in
**three places that must stay identical**. There is no auto-sync — a test is the
tripwire.

| # | Location | What |
| --- | --- | --- |
| 1 | `cerebro-mcp` → `cerebro_mcp.semantic.graph_extraction.GraphProfile` | the runtime consumer dataclass (the ultimate reference) |
| 2 | `dbt-cerebro` → `scripts/semantic/build_registry.py:703` `GRAPH_PROFILE_FIELDS` | the emitter's canonical field tuple |
| 3 | `dbt-cerebro` → `schemas/semantic_graph_catalog.schema.json` → `profiles.required` | this schema |

The coupling is enforced by [`tests/test_graph_catalog.py`](../tests/test_graph_catalog.py):

- `test_catalog_profiles_match_graphprofile_field_set` — every emitted profile's
  key set `== GRAPH_PROFILE_FIELDS` (ties emitter output ↔ emitter constant #2).
- `test_built_catalog_validates_against_committed_schema` — `jsonschema.validate(built_catalog, schema)`.
  Because the schema is `required` + `additionalProperties: false`, **any drift
  between `GRAPH_PROFILE_FIELDS` (#2) and this schema (#3) fails here.**
- `test_real_catalog_validates_against_schema_if_present` — validates the actual
  `target/semantic_graph_catalog.json` when a build exists.
- Plus determinism / self-hash / first-wins-on-duplicate-profile / section-shape tests.

> ⚠️ **`jsonschema` is an optional dependency.** The two validation tests call
> `pytest.importorskip("jsonschema")`, so if `jsonschema` is not installed they
> **silently skip**. Install it (`pip install jsonschema`) before relying on the
> schema guard locally. It is not currently pinned in CI (see below), so a
> divergence can slip through unless you run these tests with `jsonschema` present.

### CI note (accurate as of this writing)

The release workflow (`.github/workflows/build-and-release.yaml`) runs, in order:

- `build_registry.py --validate` (builds `target/semantic_graph_catalog.json`)
- `graph_gate.py` — the **blocking** gate for net-new `graph_meta_*` *authoring*
  errors (unknown kind/column, non-numeric weight, duplicate profile id, …)
- `pytest tests/test_semantic_registry.py`

`graph_gate.py` checks the authoring, **not** this JSON Schema. The
schema-structure guard is `tests/test_graph_catalog.py`, which the pinned CI step
does **not** currently invoke. Run it locally (with `jsonschema`) — or add it to
CI — when changing the contract.

## Versioning

- Emitter: `GRAPH_CATALOG_SCHEMA_VERSION = 1` (`build_registry.py:683`), stamped
  into `metadata.schema_version` (`build_registry.py:954`).
- Schema: requires `schema_version` (integer ≥ 1).
- **Any add/remove/rename of a `profiles` field is a breaking contract change →
  bump `GRAPH_CATALOG_SCHEMA_VERSION`** (consumers gate on it).
- The node-kind taxonomy [`semantic/graph_kinds.yml`](../semantic/graph_kinds.yml)
  carries its **own** independent `schema_version` — don't conflate the two.

## How to update

### A. Add / remove / rename a `profiles` field (breaking)

1. Edit the `GraphProfile` dataclass in **cerebro-mcp**
   (`cerebro_mcp/semantic/graph_extraction.py`).
2. Edit **`GRAPH_PROFILE_FIELDS`** in `scripts/semantic/build_registry.py` (add
   the key **and** the code that populates it in `build_graph_catalog`).
3. Edit this schema: update `properties.profiles.additionalProperties.required`
   **and** its `properties` block (mind the `type` — optional strings are
   `["string", "null"]`).
4. Bump `GRAPH_CATALOG_SCHEMA_VERSION` in `build_registry.py`.
5. `pip install jsonschema && pytest tests/test_graph_catalog.py` — steps 2 and 3
   diverging makes the validate test fail.
6. Rebuild and eyeball the instance:
   `python scripts/semantic/build_registry.py --target-dir target --validate`.

### B. Add a new node kind (source_kind / target_kind)

That lives in [`semantic/graph_kinds.yml`](../semantic/graph_kinds.yml), not
here — this schema only requires `source_kind`/`target_kind` to be strings. Add
the kind there, bump *its* `schema_version`, then rebuild. See
[`semantic/GRAPH_AUTHORING.md`](../semantic/GRAPH_AUTHORING.md).

### C. Add a new top-level section or field to another catalog block

Same shape as (A): update the emitter in `build_registry.build_graph_catalog`,
update this schema (respect `additionalProperties: false` and `required`), bump
`schema_version` if it changes the contract shape, and run
`tests/test_graph_catalog.py`.

### Do / Don't

- **Do** keep the schema and `GRAPH_PROFILE_FIELDS` byte-for-byte aligned on
  field names.
- **Do** commit this file; it is the published contract.
- **Don't** hand-edit `target/semantic_graph_catalog.json` — it is generated and
  self-hashed (`_catalog_hash`, `build_registry.py:798`); edits break the hash.
- **Don't** relax `additionalProperties: false` to "make validation pass" — fix
  the emitter or the schema instead.
- **Don't** rename a `profile` id without a deprecation window — a `profile` id
  becomes the catalog `edge_type` and a tool selector downstream.
